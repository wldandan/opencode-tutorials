# 2.1.1 需求收集三步法：访谈→观察→数据

> **所属章节**：第2章 需求分析与价值验证
> **阅读时长**：10分钟
> **难度等级**：⭐⭐

---

## 引言

很多LLM应用项目失败的原因，不是技术不行，而是**需求理解错了**。

某企业早期有个LLM应用项目，投入6个月开发，上线后发现根本不是用户想要的。复盘发现：需求收集只问了业务负责人，没有问一线用户，也没有看真实数据。

**需求收集是LLM应用项目的基石**，地基打不好，楼房盖不稳。本文将介绍企业实践验证过的"需求收集三步法"。

---

## 为什么需要三步法？

### 传统单一路径的问题

**常见做法**：
```
业务方：我们需要AI客服
技术：好，有什么需求？
业务方：能回答问题就行
...
3个月后上线
用户：这东西完全不好用！
```

**问题分析**：

| 问题 | 说明 | 后果 |
|-----|------|------|
| **只问领导** | 不了解一线真实痛点 | 做出来的东西没人用 |
| **没有观察** | 不了解真实工作场景 | 功能设计与实际脱节 |
| **没有数据** | 凭感觉做决策 | 效果无法量化 |

### 三步法的价值

```
┌─────────────────────────────────────────────────────────────────┐
│                       需求收集三步法                            │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│   步骤1：访谈 ────────→ 了解"他们说什么"                       │
│      （What they say）                                            │
│                                                                 │
│   步骤2：观察 ────────→ 了解"他们做什么"                       │
│      （What they do）                                             │
│                                                                 │
│   步骤3：数据 ────────→ 了解"实际发生了什么"                   │
│      （What actually happens）                                   │
│                                                                 │
│   三者交叉验证 → 真实的需求                                     │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

**核心思想**：
> 人们说的、做的、实际发生的，往往是三件不同的事。只有三步都做，才能找到真实需求。

---

## 步骤1：访谈 - 了解"他们说什么"

### 访谈的对象

不要只访谈业务负责人，要访谈三类人：

| 角色 | 能提供的信息 | 访谈重点 |
|-----|-------------|---------|
| **业务负责人** | 战略目标、资源支持、成功标准 | 为什么要做AI？期望是什么？ |
| **一线用户** | 日常工作痛点、真实需求 | 你每天做什么？哪里最麻烦？ |
| **技术接口人** | 现有系统、数据情况、技术约束 | 有什么数据？技术限制是什么？ |

### 访谈的方法

**1. 半结构化访谈**

不是完全自由聊天，也不是完全按问卷，而是：

```
准备10-15个核心问题
根据访谈情况灵活调整
深入追问感兴趣的点
```

**核心问题示例**：

```
面向业务负责人：
□ 为什么考虑用AI解决这个问题？
□ 期望AI达到什么效果？
□ 成功的标准是什么？
□ 有多少预算和时间？

面向一线用户：
□ 你每天的工作流程是什么？
□ 工作中最耗时/最麻烦的环节是什么？
□ 如果有个AI助手，你希望它帮你做什么？
□ 你觉得AI可能做不好的是什么？

面向技术接口人：
□ 现有系统有哪些？
□ 有什么数据？数据质量如何？
□ 有什么技术限制？
□ 能投入多少技术资源？
```

**2. STAR追问法**

当用户提到某个场景时，深入追问：

```
S (Situation)：什么情况下？
  "什么时候会遇到这个问题？"

T (Task)：要做什么？
  "你需要完成什么任务？"

A (Action)：怎么做的？
  "你现在是怎么处理的？"

R (Result)：结果如何？
  "这样处理的效果如何？有什么问题？"
```

### 访谈的注意事项

| ❌ 错误做法 | ✅ 正确做法 |
|-----------|-----------|
| 只访谈领导 | 三类人都要访谈 |
| 按问卷照本宣科 | 半结构化，灵活追问 |
| 只问"想要什么" | 问"痛点是什么" |
| 只问一次 | 多次访谈，持续对齐 |
| 只记录表面回答 | 深挖背后的真实需求 |

### 企业案例：智能客服项目访谈

**访谈规划**：

| 角色 | 人数 | 访谈时长 | 访谈形式 |
|-----|------|---------|---------|
| 客服主管 | 2人 | 60分钟 | 1对1 |
| 客服代表 | 10人 | 30分钟 | 1对1 |
| 技术负责人 | 2人 | 45分钟 | 1对1 |

**关键发现**：

```
客服主管说：
  - "我们需要24小时服务"
  - "希望能降低成本"
  - "提升服务质量"

客服代表说：
  - "简单问题太浪费时间"
  - "复杂问题需要专业知识"
  - "最怕用户反复问同一个问题"

技术负责人说：
  - "有10年通话记录"
  - "有知识库文档"
  - "系统比较老旧，不好改"

交叉分析发现：
  - 领导关注的是宏观指标
  - 员工关注的是日常体验
  - 数据比预期丰富，但需要处理
```

## 步骤2：观察 - 了解"他们做什么"

### 为什么要观察？

**人们说的和做的往往不一致**：

| 用户说 | 实际做 | 差异 |
|-------|-------|------|
| "我每天处理50个客户问题" | 实际30个，但每个很复杂 | 夸大数量，淡化复杂度 |
| "我按标准流程处理" | 经常跳步骤，用个人方法 | 官方流程 vs 实际做法 |
| "知识库很全" | 经常用自己的笔记 | 知识库 vs 个人经验 |

### 观察的方法

**1. 跟岗观察**

```
方法：
- 跟随一线用户工作
- 观察真实工作场景
- 记录工作流程和问题

要点：
- 不干扰正常工作
- 观察至少半天
- 记录详细笔记
```

**观察记录表**：

| 时间 | 用户行为 | 遇到的问题 | 解决方式 | 耗时 |
|-----|---------|-----------|---------|------|
| 9:05-9:15 | 接听客户电话 | 客户问题复杂，需要查询 | 问同事、查系统 | 10分钟 |
| 9:20-9:45 | 查知识库 | 找不到答案 | 搜索多个关键词 | 25分钟 |
| 9:50-10:10 | 处理工单 | 流程不熟悉 | 问主管 | 20分钟 |

**2. 关键时刻观察**

重点关注工作中的关键时刻：

```
□ 高峰时段：最忙的时候是什么样？
□ 异常处理：遇到问题怎么办？
□ 协作场景：和谁协作？怎么协作？
□ 工具使用：用什么工具？怎么用？
□ 情绪变化：什么时候最焦虑/沮丧？
```

**3. 工作环境观察**

```
物理环境：
- 办公环境（嘈杂？安静？）
- 工具设施（电脑、系统、工具）

信息环境：
- 使用的系统（几个？怎么切换？）
- 参考的资料（纸质？电子？）
- 沟通方式（电话？IM？邮件？）
```

### 企业案例：智能客服项目观察

**观察发现**：

```
发现1：知识库使用率低
  用户说：有问题查知识库
  实际观察：10次只有2次查了
  原因：知识库搜索不好用，不如问同事

发现2：复杂问题需要多人协作
  用户说：自己能处理
  实际观察：复杂问题要问2-3个同事
  原因：知识分散在不同人手里

发现3：个人笔记很重要
  用户说：知识库很全
  实际观察：每个人都有个人笔记
  原因：个人笔记更贴近实际场景
```

**观察结论**：

```
真实需求不是"更好的知识库"
而是：
1. 更好的搜索能力
2. 知识图谱式的知识关联
3. 个人经验的沉淀和共享
```

## 步骤3：数据 - 了解"实际发生了什么"

### 为什么需要数据？

**访谈和观察都是"小样本"**：
- 访谈：几十人
- 观察：几个人
- 容易受主观影响

**数据是"全样本"，更客观**：
- 真实的用户行为
- 实际的业务数据
- 客观的数量关系

### 数据收集的范围

**1. 业务数据**

| 数据类型 | 说明 | 用途 |
|---------|------|------|
| **业务指标** | KPI、工作量数据 | 量化问题严重程度 |
| **用户行为** | 点击流、操作日志 | 了解真实行为模式 |
| **服务质量** | 满意度、投诉率 | 找到痛点 |
| **成本数据** | 人力、时间、成本 | 量化价值 |

**2. 系统数据**

| 数据类型 | 说明 | 用途 |
|---------|------|------|
| **系统日志** | 操作日志、错误日志 | 了解系统使用情况 |
| **性能数据** | 响应时间、并发 | 了解技术瓶颈 |
| **内容数据** | 知识库、FAQ、工单 | 了解数据基础 |

**3. 用户反馈数据**

| 数据类型 | 说明 | 用途 |
|---------|------|------|
| **投诉数据** | 用户投诉内容 | 直接的痛点 |
| **反馈数据** | 用户反馈建议 | 改进方向 |
| **评价数据** | 满意度评分 | 效果基线 |

### 数据分析的方法

**1. 描述性分析**

```
核心问题：现状是什么？

分析内容：
□ 业务量：每天多少工单/问题？
□ 处理时长：平均处理时间？
□ 质量指标：满意度、解决率？
□ 成本指标：人力成本、时间成本？

示例发现：
- 每天2万通电话
- 平均处理时长8分钟
- 简单问题占70%
- 人力成本年5000万
```

**2. 问题分布分析**

```
核心问题：主要问题是什么？

分析方法：问题分类统计

示例发现：
问题类型      占比    平均处理时间
─────────────────────────────────
查话费        35%     2分钟
办套餐        25%     5分钟
密码问题      20%     3分钟
故障申报      15%     20分钟
其他          5%      10分钟

洞察：前3类占80%，但处理时间短
      最后一类占比少，但最耗时
```

**3. 关联分析**

```
核心问题：什么因素影响结果？

分析方法：寻找相关性

示例发现：
□ 老员工满意度更高 → 经验很重要
□ 复杂问题转接率高 → 能力不足
□ 高峰期满意度下降 → 资源不足
□ 知识库搜索多的问题解决快 → 工具重要
```

### 企业案例：智能客服项目数据分析

**数据收集**：

| 数据类型 | 数据量 | 来源 |
|---------|-------|------|
| 通话录音 | 500万条（10年） | 客服系统 |
| 工单记录 | 200万条 | 客服系统 |
| 知识库 | 5万条 | 内部系统 |
| 满意度调查 | 10万份 | CRM系统 |

**关键发现**：

```
发现1：问题分布集中
  - 前10类问题占70%
  - 前50类问题占90%
  → AI可以覆盖大部分问题

发现2：处理时长差异大
  - 简单问题：2-5分钟
  - 复杂问题：15-30分钟
  → AI应优先处理简单问题

发现3：知识库使用率低
  - 70%的问题有知识库答案
  - 但只有30%的客服主动查知识库
  → 不是知识不全，是找答案困难

发现4：高峰期压力大
  - 工作日10-16点占70%流量
  - 平均等待时间3分钟
  → AI可以缓解高峰压力
```

## 三步法：交叉验证

### 为什么要交叉验证？

```
┌─────────────────────────────────────────────────────────────────┐
│                      三步法交叉验证                              │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│   访谈：用户说"最麻烦的是查不到信息"                            │
│     ↓                                                           │
│   观察：发现用户很少用知识库搜索                                │
│     ↓                                                           │
│   数据：知识库有70%的答案，但搜索准确率只有40%                   │
│     ↓                                                           │
│   真实需求：不是"知识不全"，而是"搜索不准"                      │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 交叉验证矩阵

| 问题 | 访谈回答 | 观察结果 | 数据验证 | 真实需求 |
|-----|---------|---------|---------|---------|
| 工作量大 | "每天处理100个" | 实际50个，但都很复杂 | 日均50个工单 | 问题复杂，不是数量多 |
| 知识不够 | "知识库不全" | 很少用知识库 | 70%问题有答案 | 搜索问题，不是知识问题 |
| 成本高 | "人手不够" | 高峰期压力明显 | 高峰期人力利用率120% | 高峰期资源不足 |

### 处理冲突

当访谈、观察、数据不一致时：

| 情况 | 处理方法 |
|-----|---------|
| **访谈与观察不符** | 相信观察（行为比语言更真实） |
| **观察与数据不符** | 相信数据（大样本更客观） |
| **访谈与数据不符** | 深挖原因（为什么感觉和实际不同？） |
| **三者都不符** | 重新收集，可能遗漏了重要信息 |

## 总结

### 三步法回顾

| 步骤 | 核心问题 | 关键方法 |
|-----|---------|---------|
| **访谈** | 他们说什么？ | 三类人、半结构化、STAR追问 |
| **观察** | 他们做什么？ | 跟岗观察、关键时刻、工作环境 |
| **数据** | 实际发生了什么？ | 业务数据、系统数据、用户反馈 |
| **验证** | 真实需求是什么？ | 交叉验证、处理冲突 |

### 关键要点

```
✅ 不要只访谈领导，三类人都要访谈
✅ 不要只听用户说，要观察实际行为
✅ 不要凭感觉决策，要用数据验证
✅ 三步法交叉验证，找到真实需求
```

### 常见错误

```
❌ 只访谈业务负责人
❌ 不做现场观察
❌ 忽视数据分析
❌ 三步法各自独立，不做交叉验证
```
