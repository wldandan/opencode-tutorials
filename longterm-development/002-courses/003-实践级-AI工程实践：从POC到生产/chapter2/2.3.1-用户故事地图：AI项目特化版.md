## 引言

收集了需求，分析完了四象限，如何把这些需求组织成一个完整的解决方案？

**用户故事地图（User Story Map）**是一个强大的工具，帮助你可视化用户旅程，识别关键功能，规划产品路线图。

本文介绍企业级LLM应用项目特化的用户故事地图方法。

---

## 什么是用户故事地图？

### 核心概念

```
用户故事地图 = 可视化用户旅程 + 拆分功能优先级

核心思想：
- 从用户视角看问题
- 按时间顺序组织
- 横向：用户旅程
- 纵向：功能优先级
```

### 地图结构

```
┌─────────────────────────────────────────────────────────────────┐
│                     用户故事地图结构                             │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  用户旅程（时间轴）                                             │
│    ↓                                                            │
│  ├─→─→─→─→─→─→─→─→─→─→─→─→─→─→─→─→─→─→─→─→─→─→─→─→─→─→─→         │
│                                                                 │
│  功能优先级（纵向）                                             │
│    ↑                                                            │
│  ┌─────┬─────┬─────┬─────┐                                       │
│  │ MVP │ P2  │ P3  │ P4  │  功能卡片                         │
│  ├─────┼─────┼─────┼─────┤                                       │
│  │ 必须│ 应该│ 可以│ 未来│                                   │
│  │  做│  做│  做│  做│                                   │
│  └─────┴─────┴─────┴─────┘                                       │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

## LLM应用项目用户故事地图的特点

### 与传统项目的差异

| 维度 | 传统软件项目 | LLM应用项目 |
|-----|-------------|--------|
| **用户旅程** | 相对固定 | 需要考虑AI不确定性 |
| **功能定义** | 相对确定 | 效果驱动，需要验证 |
| **优先级** | 相对稳定 | 可能随效果变化 |
| **验收标准** | 功能验收 | 效果+功能双重验收 |

### LLM应用项目的特化要点

```
特化1：用户旅程要考虑AI的局限性
  - AI能解决什么问题？
  - AI不能解决什么问题？
  - 什么时候需要人工介入？

特化2：功能卡片要包含效果指标
  - 功能描述
  - 效果目标（如准确率）
  - 验收标准

特化3：优先级要考虑技术风险
  - 技术可行性
  - 数据可用性
  - 效果不确定性
```

## 用户故事地图的制作

### 步骤1：定义用户旅程

**核心问题**：用户在什么场景下，如何使用AI系统？

**方法：场景描述法**

```
格式：[角色] + [场景] + [目标]

示例：智能客服

场景1：话费查询
  角色：普通用户
  场景：想知道本月话费
  目标：快速获得话费信息

场景2：套餐变更
  角色：普通用户
  场景：想换个更便宜的套餐
  目标：了解套餐并完成变更

场景3：故障报修
  角色：普通用户
  场景：网络有问题需要报修
  目标：提交报修并跟踪进度

场景4：业务咨询
  角色：企业客户
  场景：想了解企业宽带业务
  目标：获取详细信息

场景5：投诉处理
  角色：不满意用户
  场景：对服务不满意要投诉
  目标：解决问题并获得反馈
```

### 步骤2：拆分功能卡片

**功能卡片格式**：

```
┌─────────────────────────────────────────────────────────────────┐
│  【功能名称】话费查询                                         │
│  【用户故事】作为用户，我想查询话费，以便知道消费情况      │
│  【AI能力】NLP理解查询意图 + 数据库查询                      │
│  【效果目标】准确率≥95%，响应时间≤2秒                       │
│  【验收标准】100个测试问题，准确率≥95%                        │
└─────────────────────────────────────────────────────────────────┘
```

### 步骤3：确定优先级

**优先级评估**：

| 因素 | 权重 | 评分标准 |
|-----|------|---------|
| **业务价值** | 40% | 5分：核心价值，3分：重要价值，1分：锦上添花 |
| **用户需求** | 30% | 5分：强烈需求，3分：一般需求，1分：可有可无 |
| **技术可行性** | 20% | 5分：成熟方案，3分：需要探索，1分：高风险 |
| **数据可用性** | 10% | 5分：数据充足，3分：需要收集，1分：数据缺乏 |

**总分计算**：
```
总分 = 业务价值×40% + 用户需求×30% + 技术可行性×20% + 数据可用性×10%

优先级：
- P1（必须）：总分≥4.0
- P2（应该）：总分3.0-3.9
- P3（可以）：总分2.0-2.9
- P4（未来）：总分<2.0
```

## LLM应用项目地图的关键要素

### 要素1：AI能力标注

```
每个功能卡片要标注AI能力：

□ 无AI（纯功能）
□ 意图识别（NLP）
□ 知识检索（RAG）
□ 流程执行（Agent）
□ 数据分析（ML）
□ 内容生成（GenAI）
□ 混合能力

示例：
【AI能力】意图识别 + 数据库查询 + 流程执行
```

### 要素2：效果目标设定

```
每个功能要设定效果目标：

□ 准确率/召回率/F1
□ 响应时间
□ 用户满意度
□ 成功率
□ 转化率

示例：
【效果目标】准确率≥75%，满意度≥70%
```

### 要素3：人工介入点

```
LLM应用项目要明确人工介入点：

□ 全AI：完全自动化
□ AI+人工：AI处理，人工审核
□ 人工+AI：AI辅助，人工决策
□ 全人工：AI不适用

示例：
【人工介入】简单问题AI处理，复杂问题转人工
```

## 总结

### 用户故事地图回顾

| 要素 | 说明 |
|-----|------|
| **用户旅程（横向）** | 按时间顺序的用户使用场景 |
| **功能卡片（纵向）** | 每个场景的功能分解 |
| **优先级** | P1-P4分优先级 |
| **AI能力** | 标注需要的AI技术 |
| **效果目标** | 设定可衡量的目标 |

### 关键要点

```
✅ 从用户视角组织旅程
✅ 按优先级排列功能
✅ 每个功能明确AI能力
✅ 设定可衡量的效果目标
✅ 标注人工介入点
```

### 常见错误

```
❌ 只从技术视角设计
❌ 优先级不清晰
❌ 没有设定效果目标
❌ 忽视AI的局限性
❌ 没有考虑人工介入
```
