# 2.1.3 技术需求转换：业务目标→技术指标→验收标准

> **所属章节**：第2章 需求分析与价值验证
> **阅读时长**：10分钟
> **难度等级**：⭐⭐⭐

---

## 引言

业务方说"我们要一个智能客服"，技术方直接开始开发，三个月后上线，结果业务方说"这不是我想要的"。

这是LLM应用项目最常见的失败原因之一：**业务需求没有正确转换为技术需求**。

本文将介绍企业实践验证过的"技术需求转换三步法"：从业务目标到技术指标，再到验收标准。

---

## 为什么需要转换？

### 常见的问题

**业务语言 vs 技术语言**

```
业务方说："要一个智能的客服系统"
技术方听到："做一个问答系统"
实际做出来：一个规则匹配的问答机器人
业务方体验："这不智能啊！"

问题：业务方的"智能" ≠ 技术方的"问答"
```

**期望 vs 能力**

```
业务期望："像人一样聪明"
技术现实：大模型有局限性

问题：没有管理期望，设定不切实际的目标
```

**目标 vs 指标**

```
业务目标："提升用户体验"
技术实现：做了很多功能

问题：体验提升如何衡量？用什么指标？
```

## 第一步：业务目标 → 技术指标

### 转换原则

**SMART原则**：

| 原则 | 说明 | 示例 |
|-----|------|------|
| **Specific** | 具体的 | 不说"智能"，说"能回答90%的问题" |
| **Measurable** | 可衡量的 | 用数字指标，不用"感觉" |
| **Achievable** | 可实现的 | 基于技术现状，不盲目乐观 |
| **Relevant** | 相关的 | 指标与业务目标相关 |
| **Time-bound** | 有时限的 | 明确时间范围 |

### 业务目标到技术指标的映射

| 业务目标 | 技术指标 | 设定方法 |
|---------|---------|---------|
| **提升处理效率** | 处理时长、并发量 | 基于现状提升 |
| **降低人力成本** | AI解决率 | 能自动化多少 |
| **提升用户满意度** | 满意度评分 | 基于调研设定 |
| **减少错误率** | 准确率、召回率 | 基于业务容忍度 |

### 实际案例：智能客服

**业务目标**：
```
1. 降低客服人力成本50%
2. 提升24小时服务能力
3. 用户满意度不低于人工
```

**转换为技术指标**：

| 业务目标 | 技术指标 | 设定依据 |
|---------|---------|---------|
| 降低成本50% | AI解决率≥70% | 70%×10万/天 = 7万/天 ≈ 降低50%人力 |
| 24小时服务 | 系统可用性≥99.9% | 全年停机<8.76小时 |
| 满意度不降低 | 满意度≥人工的90% | 人工满意度80% → AI≥72% |

### 企业实践：指标设定的三大陷阱

**陷阱1：指标过高，不可实现**

```
❌ 错误：
"准确率要达到95%"

问题：当前技术最高80%
结果：项目无法验收

✅ 正确：
"准确率≥75%，持续优化到80%+"
基于：当前技术可达，有提升空间
```

**陷阱2：指标过低，没价值**

```
❌ 错误：
"能回答问题就行"

问题：太模糊，无法验收

✅ 正确：
"简单问题（70%）准确率≥85%"
"复杂问题（30%）准确率≥60%"
分场景设定，有明确价值
```

**陷阱3：指标单一，不全面**

```
❌ 错误：
只用"准确率"一个指标

问题：准确率高的回答可能用户不满意

✅ 正确：
多维度指标
- 准确率（技术指标）
- 满意度（用户体验）
- 解决率（业务价值）
- 响应时间（性能）
```

## 第二步：技术指标 → 验收标准

### 为什么要验收标准？

```
指标：准确率≥75%
问题：怎么算？什么时候测？

验收标准：
- 数据集：测试集1000个问题
- 测试方法：离线评估
- 通过标准：≥75%算通过
```

### 验收标准模板

```
┌─────────────────────────────────────────────────────────────────┐
│                      验收标准模板                              │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  【指标名称】智能客服准确率                                     │
│                                                                 │
│  【指标定义】AI回答与标准答案的一致性比例                        │
│                                                                 │
│  【测试方法】                                                   │
│  1. 准备测试数据集：1000个真实用户问题                          │
│  2. 标准答案：2名专家标注，取一致答案                            │
│  3. 测试方式：离线批量测试                                      │
│  4. 评估方法：自动比对 + 人工抽检10%                            │
│                                                                 │
│  【通过标准】                                                   │
│  - 整体准确率≥75%                                              │
│  - 简单问题准确率≥85%                                         │
│  - 复杂问题准确率≥60%                                         │
│                                                                 │
│  【测试环境】                                                   │
│  - 测试时间：□POC □MVP □试点                                  │
│  - 测试数据：□历史 □新收集 □混合                             │
│  - 测试条件：□离线 □在线 □灰度                              │
│                                                                 │
│  【验收时间】                                                  │
│  - 测试周期：1周                                             │
│  - 结果确认：测试后3个工作日                                  │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 各阶段的验收标准

**不同阶段，不同标准**：

| 阶段 | 验收重点 | 通过标准 | 示例 |
|-----|---------|---------|------|
| **POC** | 技术可行性 | 达到预期60%+ | 准确率目标70%，实际≥42% |
| **MVP** | 用户接受度 | 满意度≥65% | 灰度测试用户反馈 |
| **试点** | 业务价值 | 核心指标达标 | AI解决率、成本、满意度 |
| **规模** | 可复制性 | 成功复制到N个场景 | 推广进度、质量稳定性 |

## 第三步：分场景验收标准

### 为什么要分场景？

**单一指标的局限**：

```
用"准确率"一个指标评价所有问题：
❌ 简单问题和复杂问题一个标准
❌ 不同业务类型一个标准
❌ 没有考虑用户体验

✅ 分场景设定标准：
- 简单问题：85%（用户期望高）
- 复杂问题：60%（用户理解难度）
- 极复杂问题：30%（转人工）
```

### 分场景方法

**按问题复杂度**：

| 场景 | 占比 | 准确率目标 | 权重 |
|-----|------|-----------|------|
| 简单 | 50% | ≥90% | 0.3 |
| 中等 | 30% | ≥75% | 0.3 |
| 复杂 | 15% | ≥60% | 0.2 |
| 极复杂 | 5% | ≥30% | 0.2 |

**综合评估**：
```
加权准确率 = 90%×0.3 + 75%×0.3 + 60%×0.2 + 30%×0.2 = 68%
```

**按业务类型**：

| 业务类型 | 验收标准 | 说明 |
|---------|---------|------|
| **查询类** | 准确率≥90% | 查话费、余额等，要求高 |
| **办理类** | 成功率≥80% | 办套餐、改套餐等 |
| **咨询类** | 相关性≥75% | 问题解答，允许不完美 |
| **投诉类** | 转人工率≥95% | 敏感问题，优先转人 |

### 企业案例：分场景验收标准

**某运营商智能客服**：

```
场景1：话费查询（30%）
  准确率≥95%
  响应时间≤2秒
  权重：0.2

场景2：套餐办理（25%）
  成功率≥85%
  完成时间≤3分钟
  权重：0.3

场景3：故障排查（20%）
  定位准确率≥70%
  解决建议相关度≥75%
  权重：0.3

场景4：投诉处理（15%）
  转人工率≥90%
  响应时间≤1分钟
  权重：0.1

场景5：其他（10%）
  F1≥70%
  权重：0.1

综合评分 = Σ(场景得分 × 权重)
```

## 完整转换案例

### 案例：智能文档处理系统

**业务目标**：
```
1. 文档处理效率提升5倍
2. 处理准确率不低于人工
3. 降低人力成本60%
```

**第一步：转换为技术指标**

| 业务目标 | 技术指标 | 设定依据 |
|---------|---------|---------|
| 效率提升5倍 | 处理时长≤人工的20% | 人工30分钟/份 → AI≤6分钟 |
| 准确率不低于人工 | 准确率≥人工的95% | 人工95% → AI≥90% |
| 降低成本60% | AI自动化率≥80% | 80%自动化 → 节省4/5人力 |

**第二步：设定验收标准**

```
指标1：处理时长
- 测试集：100份文档
- 测试方法：端到端计时
- 通过标准：平均≤6分钟，95%≤10分钟

指标2：准确率
- 测试集：500份标注文档
- 评估方法：字段级F1≥90%
- 通过标准：整体F1≥90%

指标3：自动化率
- 测试方法：无人工干预处理比例
- 通过标准：≥80%
```

**第三步：分场景标准**

| 文档类型 | 占比 | 时长目标 | 准确率目标 | 权重 |
|---------|------|---------|-----------|------|
| 简单表格 | 40% | ≤3分钟 | ≥95% | 0.2 |
| 复杂表格 | 30% | ≤8分钟 | ≥85% | 0.4 |
| 混合文档 | 20% | ≤10分钟 | ≥80% | 0.3 |
| 非标准 | 10% | 转人工 | - | 0.1 |

## 总结

### 三步法回顾

```
业务目标 → 技术指标 → 验收标准
  ↓         ↓          ↓
 "要什么"  "怎么衡量"  "怎么验收"
 量化      数据化      可测试化
```

### 关键要点

```
✅ 业务目标要量化（SMART原则）
✅ 技术指标要可衡量
✅ 验收标准要可测试
✅ 分场景设定更精准
✅ 不同阶段不同标准
```

### 常见错误

```
❌ 业务目标模糊："要一个智能的XX"
❌ 技术指标过高/过低
❌ 验收标准不清晰
❌ 用单一指标评价
❌ 不分场景，一刀切
```

3. **细化**：为你的项目设定分场景的验收标准
